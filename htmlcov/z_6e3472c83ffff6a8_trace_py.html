1"""Live-step-trace renderer: aggregated Components-V2 step messages (ADR-0017). 

2 

3The bridge's :class:`~calfcord.bridge.mention_handler.MentionHandler` drains a 

4run's ``stream()`` and, for every non-A2A 

5:class:`~calfcord.bridge.step_events.StepEvent`, calls 

6:meth:`StepTraceRenderer.on_step`; a ``finally`` always calls 

7:meth:`StepTraceRenderer.finish`. :meth:`StepTraceRenderer.on_consult` folds in 

8the bridge's OWN annotation of a consult — same trace, no ``StepEvent``, since 

9the A2A dispatcher intercepts both halves before ``on_step`` is reached. 

10 

11Steps are aggregated into ONE growing, persistent **Components-V2** message per 

12persona *segment*: the first renderable step posts the message, later steps are 

13appended and the message is edited in place. A new message starts only when 

14 

15* the **persona changes** — a webhook edit cannot change ``username``/ 

16 ``avatar_url``, so a handoff (or a peer emitter) opens a fresh message under 

17 the new identity; or 

18* the **4000-char v2 cap** would overflow — rollover to a fresh message, same 

19 persona, full trace preserved (no elision). 

20 

21Nothing is ever deleted: the aggregate(s) persist as the turn's visible trace. 

22Because a v2 message carries no ``content`` (only components), the bridge's 

23history fetcher excludes these from an agent's ``message_history`` — the 

24model's tool memory rides the separate transcript replay, so the display never 

25double-counts (the ADR-0016 invariant, unchanged). 

26 

27**Writer task (leading-edge throttle).** ``on_step`` never touches Discord: it 

28appends (or resolves) rows on the correlation's entry and sets a wake event. 

29A per-entry writer task flushes every dirty segment (post, then in-place edits), 

30then waits :data:`_MIN_EDIT_INTERVAL_SECONDS` before the next flush — so an 

31idle stream renders each step immediately, a burst coalesces into ≤1 edit per 

32interval, and a Discord stall back-pressures only the trace display, never the 

33stream drain. :meth:`finish` signals the writer, which does a final flush and 

34exits — pending content is never lost, and the interval wait is interrupted so 

35the terminal reply (behind ``finish`` in the handler) is not delayed. 

36 

37**Persona** is resolved per step. For agent-authored steps (``agent_message``, 

38``handoff``) the emitter's persona is used so a peer after a handoff stamps its 

39own identity. For tool steps (``tool_call``, ``tool_result``) the **acting 

40agent** — the agent currently in control of the run — is used instead, because a 

41tool is a utility, not a conversational participant. The acting agent is tracked 

42by the handler (initialised to the mention target, updated on handoff) and passed 

43in via ``acting_agent``. 

44 

45**Rows are values** (ADR-0024): a ``tool_call`` opens a PENDING row and its 

46result RESOLVES that same row in place, keyed by ``tool_call_id`` — results 

47arrive in completion order, so a row is never found by position. **The seal** 

48(ADR-0025) closes the trace with the run's outcome, taken from the stream's 

49terminal. 

50 

51**Failure semantics.** Every Discord call is best-effort 

52(:func:`_best_effort_trace`): a failed send/edit must never crash the run or 

53affect the terminal reply. A failed *post* leaves the segment dirty so the next 

54wake retries it — and the final flush, which has no next wake, retries once 

55inline. A 

56failed *edit* is dropped without a retry (avoiding a hot loop on e.g. a deleted 

57message); every edit re-renders the segment's full current body, so the next 

58append heals the gap — EXCEPT on the final flush, which has no next append and 

59therefore retries once, because the seal is always that last append and a 

60frozen ``◐`` is the one outcome ADR-0025 rules out. A non-Discord error inside a flush is caught and logged 

61by the writer loop itself, so it cannot unwind ``finish``. A mid-run bridge 

62restart strands only a persistent partial trace (cosmetic; no state to recover) 

63— the one case the seal cannot rescue, since the run dies with the bridge. 

64 

65**Typing** is disabled for now — the fire call is commented out; the notifier is 

66still accepted (dormant) for a one-line re-enable. 

67"""

109"""Discord's Components-V2 hard cap — per ``TextDisplay`` AND per whole message 

110(the sum of all text across every container/text display). Verified against the

112client-side (``LayoutView.content_length()`` is a helper, not a guard). This is a 

113*message* limit, so the segment owns it; :data:`~calfcord.bridge.trace_rows._ROW_GROWTH_RESERVE` 

114is the row's concern."""

118"""Leading-edge throttle: the minimum spacing between two flushes of one run's

150def _is_pending(row: TraceRow) -> TypeGuard[ToolRow | ConsultRow]: 

151 """Whether ``row`` still has a growth reservation booked against it. 

152 

153 Only the keyed rows resolve; prose, handoffs, and the seal are born final. 

154 

155 A ``TypeGuard`` rather than a plain ``bool`` because callers rely on the 

156 narrowing it performs: ``_seal_entry`` does ``replace(row, state=...)``, which 

157 is only valid on the two variants that HAVE a state. The guard is what makes 

158 that safety checkable instead of merely true. 

159 """ 

160 return isinstance(row, ToolRow | ConsultRow) and row.state == "pending"

163@dataclass(slots=True) 

164class _Segment: 

165 """One Discord message of a step trace: a contiguous run of rows sharing a 

166 persona, capped at the v2 whole-message limit. 

167 

168 Rows are VALUES (ADR-0024) — resolving one replaces it and re-renders. That 

169 is affordable because every flush re-sends the whole body anyway, so 

170 revising an earlier row costs exactly what appending one costs. 

171 

172 ``message_id`` stays ``None`` until the first successful post. ``dirty`` 

173 marks content Discord has not shown yet; the writer clears it before 

174 flushing (a failed POST re-marks it — unposted content must survive to the 

175 next wake; a failed EDIT does not — the next append re-marks anyway and 

176 every edit re-renders the full body). 

177 """ 

178 

179 persona: Persona 

180 rows: list[TraceRow] = field(default_factory=list) 

181 message_id: int | None = None 

182 dirty: bool = False 

183 

184 @property 

185 def pending(self) -> int: 

186 return sum(1 for row in self.rows if _is_pending(row)) 

187 

188 def _join(self, rows: list[TraceRow]) -> str: 

189 return _ROW_JOINER.join(render_row(row) for row in rows) 

190 

191 def fits(self, row: TraceRow) -> bool: 

192 """Whether ``row`` fits — INCLUDING the worst-case growth of every row 

193 that has yet to resolve. 

194 

195 A segment that has been posted cannot be re-split, so a resolve must 

196 never be the thing that overflows it. Each pending row books 

197 ``_ROW_GROWTH_RESERVE``; the booking is released as it resolves, so the 

198 cost is transient and proportional to the fan-out actually in flight. 

199 """ 

200 booked = _ROW_GROWTH_RESERVE * (self.pending + (1 if _is_pending(row) else 0)) 

201 return len(self._join([*self.rows, row])) + booked <= _V2_TEXT_LIMIT 

202 

203 def append(self, row: TraceRow) -> int: 

204 """Append ``row``; returns its index, which is stable — rows are never 

205 removed or reordered, only replaced.""" 

206 self.rows.append(row) 

207 self.dirty = True 

208 return len(self.rows) - 1 

209 

210 def replace(self, index: int, row: TraceRow) -> None: 

211 self.rows[index] = row 

212 self.dirty = True 

213 

214 def body(self) -> str: 

215 """The segment's rendered body, hard-capped so it is ALWAYS postable. 

216 

217 The cap is a backstop that :meth:`fits` should keep unreachable, and it 

218 exists because the alternative failure is silent and permanent: an 

219 over-cap body is a 400, which the best-effort send swallows to a 

220 WARNING, leaving the segment dirty and retried on every wake — forever, 

221 and never rendered. 

222 

223 Reaching it means the growth-reservation invariant broke, so it warns: 

224 this is the one place that can silently drop real trace. 

225 """ 

226 rendered = self._join(self.rows) 

227 if len(rendered) > _V2_TEXT_LIMIT: 

228 logger.warning( 

229 "trace: segment over the v2 cap (%d chars across %d rows); truncating — " 

230 "the fits() growth reservation did not hold", 

231 len(rendered), 

232 len(self.rows), 

233 ) 

234 return rendered[:_V2_TEXT_LIMIT]

239 """Per-correlation state for one in-flight run's trace. 

240 

241 ``thread_id`` routes the messages INTO the thread the wire originated in

311async def _best_effort_trace[T](coro: Awaitable[T], *, channel_id: int) -> T | _Failed: 

312 """Await a best-effort trace send/edit, swallowing the usual Discord 

313 failures so the trace can never crash the run. Returns the call's result, 

314 or :data:`_FAILED` if it was swallowed. 

315 

316 ``NotFound`` (already gone) is DEBUG; ``Forbidden`` and the broader 

317 ``DiscordException`` (which also funnels the sibling ``RateLimited``, NOT a 

318 subclass of ``HTTPException``) are WARNING. ``CancelledError`` is a 

319 ``BaseException`` and is intentionally not caught, so shutdown stays clean. 

320 

321 ``aiohttp.ClientError`` is caught too, and it is not a theoretical case: 

322 a dropped keep-alive surfaces as ``ServerDisconnectedError``, which 

323 discord.py does NOT wrap in a ``DiscordException`` — it comes straight from 

324 the transport. Left uncaught it escapes to the writer loop, which logs it and 

325 moves on, but by then ``_flush`` has cleared ``dirty``: the segment is clean, 

326 never retried, and its content is silently gone. Returning :data:`_FAILED` 

327 instead routes a transport blip into the same "post failed, retry" path as 

328 every other transient failure. 

329 """ 

330 try: 

331 return await coro 

332 except discord.NotFound: 

333 logger.debug("trace: call hit NotFound channel_id=%d (already gone)", channel_id) 

334 except discord.Forbidden: 

335 logger.warning("trace: call Forbidden channel_id=%d", channel_id) 

336 except discord.DiscordException as e: 

337 logger.warning( 

338 "trace: call failed channel_id=%d status=%s: %s", 

339 channel_id, 

340 getattr(e, "status", None), 

341 e, 

342 ) 

343 except aiohttp.ClientError as e: 

344 logger.warning("trace: call hit a transport error channel_id=%d: %r", channel_id, e) 

345 return _FAILED

348class StepTraceRenderer: 

349 """Aggregates run steps into throttled, edited-in-place v2 trace messages. 

350 

351 Satisfies the ``StepTraceRendererLike`` protocol the 

352 :class:`~calfcord.bridge.mention_handler.MentionHandler` injects. Construct 

353 one per RENDERING SURFACE from the REST-only persona sender and (optionally) a 

354 typing notifier (currently dormant): the bridge has two — the human's thread, 

355 and the A2A audit thread each consulted agent's trace renders into 

356 (ADR-0026). They MUST be separate instances because ``_entries`` is keyed by 

357 ``correlation_id`` and both surfaces see the same one for a given turn. 

358 

359 ``min_edit_interval`` exists for tests, and as the one lever if a surface 

360 proves too chatty for its webhook's bucket (the A2A channel funnels every 

361 thread through ONE webhook — see the runbook in ``docs/a2a-threads.md``); 

362 neither production instance passes it, so both take the default. 

363 """

386 async def on_step(self, step: StepEvent, dest: Destination, *, acting_agent: str) -> None: 

387 """Fold one step into the correlation's trace — no Discord I/O here. 

388 

389 A ``tool_call`` opens a PENDING row; its ``tool_result`` resolves that 

390 same row in place, so one tool is one row rather than a called/returned 

391 pair. A ``handoff`` and an ``agent_message`` append final rows. A step 

392 that renders nothing touches nothing (no entry, no writer task). The 

393 writer task (created lazily with the entry) is woken to flush. 

394 

395 ``acting_agent`` is the agent currently in control of the run (the 

396 mention target initially, updated on handoff). It is the persona for 

397 ``tool_call``/``tool_result`` steps so tool trace rows don't appear 

398 under the tool's own identity; ``agent_message``/``handoff`` steps keep 

399 ``step.emitter`` — the genuine peer-emitter cases. 

400 """ 

401 # Tools are utilities, not conversational participants — their trace rows 

402 # appear under the calling agent's persona. Agent-authored steps 

403 # (messages, handoffs) keep the emitter's persona so a peer after a 

404 # handoff stamps its own identity. 

405 persona_name = acting_agent if step.kind in ("tool_call", "tool_result") else step.emitter 

406 persona = persona_for(persona_name) 

407 

408 match step.kind: 

409 case "tool_call": 

410 entry = self._entry_for(step.correlation_id, dest) 

411 subject, detail = _summarise_args(step.args or {}) 

412 key = step.tool_call_id 

413 row = ToolRow(key=key or "", name=_plain(step.name or "?"), subject=subject, detail=detail) 

414 if key is None: 

415 # Unkeyable, so never registered: `or ""` would alias EVERY 

416 # null id onto one key, and the second call would overwrite 

417 # the first — the first result then resolving the SECOND row, 

418 # rendering tool A's outcome on tool B's line. An unkeyed row 

419 # can never be resolved, so the seal interrupts it honestly. 

420 logger.warning("trace: tool_call %r has no tool_call_id; its row cannot resolve", step.name) 

421 self._append(entry, persona, row) 

422 else: 

423 self._append_keyed(entry, persona, row) 

424 entry.started[key] = self._now() 

425 entry.tool_count += 1 

426 case "tool_result": 

427 entry = self._entry_for(step.correlation_id, dest) 

428 self._resolve_tool(entry, persona, step) 

429 case "handoff": 

430 entry = self._entry_for(step.correlation_id, dest) 

431 self._append( 

432 entry, 

433 persona, 

434 HandoffRow( 

435 target=_plain((step.target or "").removeprefix("/")), 

436 reason=_plain(step.reason or ""), 

437 ), 

438 ) 

439 case "agent_message": 439 ↛ 446line 439 didn't jump to line 446 because the pattern on line 439 always matched

440 text = step.text.strip() 

441 if not text: 

442 return # renders nothing → touches nothing 

443 entry = self._entry_for(step.correlation_id, dest) 

444 for chunk in _chunk_text(text, _V2_CHUNK): 

445 self._append(entry, persona, ProseRow(text=chunk)) 

446 case _: 

447 # Defensive: a future calfkit kind (e.g. ``agent_thinking``) must 

448 # render nothing rather than fault the drain — logged so the gap 

449 # is visible. 

450 logger.warning("trace: no row for step kind %r; rendering nothing", step.kind) 

451 return 

452 # Typing disabled for now — re-enable by uncommenting (the notifier is 

453 # still wired through the gateway, just dormant): 

454 # if self._typing is not None: 

455 # self._typing.fire(entry.thread_id or entry.channel_id) 

456 entry.wake.set()

458 def _resolve_tool(self, entry: _Entry, persona: Persona, step: StepEvent) -> None: 

459 """Resolve a ``tool_call``'s row in place from its result. 

460 

461 The row may sit in an OLDER segment than the one currently being 

462 appended to (cap rollover mid-flight) — that is fine: every segment 

463 carries its own ``message_id``, so it simply goes dirty and is edited. 

464 """ 

465 key = step.tool_call_id 

466 state = _RESULT_STATE.get(step.outcome) 

467 if state is None: 467 ↛ 471line 467 didn't jump to line 471 because the condition on line 467 was never true

468 # A future calfkit outcome. Degrade to `failed` (visible, honest 

469 # about not having succeeded) rather than KeyError the step, which 

470 # would leave the row pending until the seal called it interrupted. 

471 logger.warning("trace: unknown tool outcome %r; rendering as failed", step.outcome) 

472 state = "failed" 

473 note = _plain(step.text) 

474 located = entry.locate.get(key) if key is not None else None 

475 if located is None: 

476 # Orphan: no call row was ever seen for this id. Append a resolved 

477 # row rather than raise — the drain's contract is that the render 

478 # path can never fault the turn, and a dropped result would also 

479 # silently under-count the turn's tools. No duration: there was 

480 # nothing to measure from. 

481 subject, detail = _summarise_args(step.args or {}) 

482 self._append( 

483 entry, 

484 persona, 

485 ToolRow( 

486 key=key or "", 

487 name=_plain(step.name or "?"), 

488 subject=subject, 

489 detail=detail, 

490 state=state, 

491 note=note, 

492 ), 

493 ) 

494 return 

495 segment, index = located 

496 row = segment.rows[index] 

497 assert isinstance(row, ToolRow) # locate only ever holds keyed rows 

498 started = entry.started.pop(key, None) if key is not None else None 

499 elapsed_ms = int((self._now() - started) * 1000) if started is not None else None 

500 segment.replace(index, replace(row, state=state, note=note, elapsed_ms=elapsed_ms)) 

501 # Retire the key. A resolved row books ZERO growth reserve, so letting a 

502 # second result re-resolve it would grow the segment into budget nobody 

503 # reserved — past fits() and into the silent hard cap. A duplicate id now 

504 # lands as an orphan row instead, which is visible rather than corrupting. 

505 entry.locate.pop(key, None) # type: ignore[arg-type]

507 async def on_consult( 

508 self, 

509 key: str, 

510 peer: str, 

511 thread_url: str | None, 

512 dest: Destination, 

513 *, 

514 correlation_id: str, 

515 persona_name: str, 

516 inline: bool = False, 

517 request_preview: str = "", 

518 ) -> None: 

519 """Open a PENDING consult row — the bridge's own annotation on the turn. 

520 

521 Not a run step: the A2A dispatcher intercepts both halves of a 

522 ``message_agent`` call before ``on_step`` is ever reached, so the 

523 correlation and persona are passed explicitly. It shares the trace 

524 deliberately — a consult is part of the same narrative, so under an 

525 unchanged persona it flows inline rather than opening a second message. 

526 

527 ``key`` is the consult's ``tool_call_id``, which every A2A projection 

528 already carries — so the row resolves through exactly the same index as a 

529 tool row. 

530 

531 Per ADR-0020 the exchange itself is private: this shows only THAT the 

532 consult happened and where to read it. The exception is a nested consult

592 async def seal(self, correlation_id: str, *, faulted: bool) -> None: 

593 """Close the correlation's trace with the run's outcome (ADR-0025). 

594 

595 Driven by the terminal (``RunCompleted``/``RunFailed``) the drain 

596 already receives — ``finish`` cannot do this, because it runs in a 

597 ``finally`` around the drain while the fault only surfaces afterwards in 

598 ``_await_terminal``. A seal written there would render ``4 tools · 

599 12.3s`` on a crashed turn. 

600 

601 Three jobs, all lookups on the one row index: 

602 

603 * resolve every still-``pending`` row to ``interrupted`` — a fault 

604 strands rows mid-flight, and the bridge is alive and knows; 

605 * append the :class:`SealRow`; 

606 * mark the entry sealed so ``finish`` does not seal it again. 

607 

608 A no-op for a correlation that never produced a renderable step: sealing 

609 must not conjure a trace out of nothing. 

610 """ 

611 entry = self._entries.get(correlation_id) 

612 if entry is None or entry.sealed: 

613 return 

614 self._seal_entry(entry, outcome="faulted" if faulted else "ok") 

615 entry.wake.set()

638 async def finish(self, correlation_id: str) -> None: 

639 """Flush pending content and retire the correlation's writer. 

640 

641 Seals defensively first: an entry that reached here unsealed means the 

642 drain raised, the stream broke, or calfkit violated its 

643 exactly-one-terminal contract. Sealing it ``interrupted`` keeps a frozen 

644 ``◐`` — a trace permanently asserting "still running" — off the channel. 

645 

646 Runs on success AND fault (the handler calls it in a ``finally``). Pops 

647 the entry, signals the writer — interrupting its interval wait so the 

648 terminal reply behind this call is not delayed — and awaits its final 

649 flush. The trace messages persist; nothing is deleted. A no-op for a 

650 correlation that never produced a renderable step. 

651 """ 

652 entry = self._entries.pop(correlation_id, None) 

653 if entry is None: 

654 return 

655 if not entry.sealed: 

656 logger.warning( 

657 "trace: correlation_id=%s finished unsealed (no stream terminal); sealing as interrupted", 

658 correlation_id, 

659 ) 

660 # NOT "faulted": no terminal was seen, so the outcome is genuinely 

661 # unknown — result() may still have returned a reply. 

662 self._seal_entry(entry, outcome="interrupted") 

663 entry.finished.set()

668 def _append(self, entry: _Entry, persona: Persona, row: TraceRow) -> tuple[_Segment, int]: 

669 """Append one row, opening a new segment when needed. 

670 

671 Only the last segment is appendable (chronological order); a persona 

672 change or a cap overflow closes it. ``persona_for`` is deterministic, so 

673 dataclass equality is a correct same-identity check. 

674 

675 A row that fits nowhere — not even a fresh segment — is still appended: 

676 there is nowhere else for it to go, and ``_Segment.body`` caps the result 

677 so the segment stays postable. Dropping it instead would lose trace, and 

678 posting an over-cap body would strand the segment dirty forever. 

679 """ 

680 segment = entry.segments[-1] if entry.segments else None 

681 if segment is None or segment.persona != persona or not segment.fits(row): 

682 segment = _Segment(persona=persona) 

683 entry.segments.append(segment) 

684 return segment, segment.append(row)

690 async def _write_loop(self, entry: _Entry) -> None: 

691 """The entry's single flusher: wake → flush → spacing → repeat. 

692 

693 The spacing wait is interruptible ONLY by ``finished`` (never by new 

694 content), so two flushes are always ≥ the interval apart; an idle writer 

695 parked on ``wake`` flushes a fresh step immediately (leading edge). A 

696 flush bug (non-Discord) is contained here — logged, loop continues — so 

697 it can never unwind ``finish`` or cost the terminal reply. 

698 """ 

699 while True: 

700 await entry.wake.wait() 

701 entry.wake.clear() 

702 try: 

703 await self._flush(entry) 

704 except Exception: 

705 logger.exception("trace: flush failed; continuing") 

706 # `and not wake` is load-bearing: a flush awaits a full Discord 

707 # round-trip, and anything appended DURING it re-sets `wake`. 

708 # Exiting on `finished` alone would drop that content permanently — 

709 # the entry is already popped and the writer is gone, so the message 

710 # would stay as-is forever. The seal is the likeliest casualty: it 

711 # lands exactly when the writer is busy, so the trace would freeze 

712 # mid-`◐`, which is what ADR-0025 exists to prevent. 

713 # 

714 # It cannot spin: only an append sets `wake`, and after `finish` the 

715 # drain is done, so at most one more pass runs. A failed POST 

716 # re-marks `dirty` WITHOUT setting `wake`, so a persistently failing 

717 # send still exits after one retry rather than hot-looping. 

718 if entry.finished.is_set() and not entry.wake.is_set(): 

719 return 

720 if self._min_edit_interval > 0: 

721 with contextlib.suppress(TimeoutError): 

722 await asyncio.wait_for(entry.finished.wait(), self._min_edit_interval) 

723 else: 

724 await asyncio.sleep(0) # tests: still yield so the loop can't starve

726 async def _flush(self, entry: _Entry) -> None: 

727 """Render every dirty segment to Discord: first post, then in-place edits. 

728 

729 The body snapshot is taken before the await; a block appended mid-call 

730 re-marks the segment dirty and is carried by the next flush. Iterates a 

731 copy — ``on_step`` may append a NEW segment while an older one is being 

732 sent (it is picked up on the next wake, which that append also set). 

733 

734 **A later segment is never posted before an earlier one lands.** Discord 

735 orders by post time and this trace deletes nothing, so posting out of 

736 order inverts the turn permanently — the reader sees the handoff before 

737 the work that caused it. An unposted earlier segment therefore holds back 

738 the ones behind it; it stays dirty and retries on the next wake. Only 

739 POSTS are held: an edit cannot reorder anything. 

740 """ 

741 blocked = False 

742 for segment in list(entry.segments): 

743 if blocked and segment.message_id is None: 

744 continue # still dirty → retried once the earlier segment lands 

745 if not segment.dirty: 

746 continue 

747 # Render BEFORE clearing dirty: if the render raises, the writer 

748 # loop logs it and continues — and a segment already marked clean 

749 # would never be retried, so its content would be gone silently. 

750 view = _build_segment_view(segment.body(), accent_for(segment.persona.name)) 

751 segment.dirty = False 

752 if segment.message_id is None: 

753 sent = await self._post(entry, segment, view) 

754 if isinstance(sent, _Failed): 

755 # Unposted content must not be lost: retry on the next wake 

756 # (or the final flush). Bounded — one attempt per wake. 

757 segment.dirty = True 

758 blocked = True # nothing behind this may post ahead of it 

759 else: 

760 segment.message_id = sent.id 

761 else: 

762 await self._edit(entry, segment, view)

764 async def _post(self, entry: _Entry, segment: _Segment, view: discord.ui.LayoutView) -> SentMessage | _Failed: 

765 """Post one segment, retrying ONCE on a final flush. 

766 

767 The mirror of :meth:`_edit`, and for the mirror-image reason. A failed 

768 post normally retries on the next wake — but it sets ``dirty`` WITHOUT 

769 setting ``wake`` (that is what stops a hot loop), so on the final flush 

770 the writer exits and **there is no next wake**. One transient 500 would 

771 then drop the whole segment, seal included, leaving a trace that reads 

772 "still running" forever. 

773 

774 Reachable whenever the seal's segment is unposted at ``finish``: a cap 

775 rollover the seal opens, or a run that ends before the first flush. 

776 """

779 return self._persona_sender.send_components( 

780 persona=segment.persona, 

781 channel_id=entry.channel_id, 

782 view=view, 

783 thread_id=entry.thread_id, 

784 ) 

785 

786 sent = await _best_effort_trace(_attempt(), channel_id=entry.channel_id) 

787 if not isinstance(sent, _Failed) or not entry.finished.is_set(): 

788 return sent 

789 logger.debug("trace: retrying the final post channel_id=%d", entry.channel_id) 

790 return await _best_effort_trace(_attempt(), channel_id=entry.channel_id)
